# task-flow

Плагин Claude Code: дисциплинированный **пофазовый flow ведения задачи** плюс
**детерминированный CI-гейт**, на который он опирается, — и скилл `decompose`,
который питает оба. Три скилла, один продукт.

## Пайплайн

```
decompose → task → ci-gate
```

`decompose` режет эпик/фичу/спеку на задачи; каждая задача проходит flow
`task` от начала до конца; `ci-gate` — детерминированный «пол», через который
проходит каждый merge.

## Скиллы

### `decompose` — эпик/фича/спека → декомпозиция на задачи
Превращает одну крупную единицу работы — вольное текстовое описание фичи,
существующий эпик трекера `<TASK-ID>` или спека/design-документ — в
well-formed, связанные зависимостями задачи с полным контрактом из **6
полей** (`name`, `context`, `requirements`, `dod` с `truths`, `story_points`,
`depends_on`), графом `depends_on` и вычисленными волнами параллелизма:

```
0 ingest → 1 requirements → 2 decompose → 3 enrich → 4 graph/waves
→ 5 QA (субагент в чистом контексте) → 6 MD-драфт → 7 tracker sync*
```

`*` опционально: фаза 7 запускается только после явного одобрения драфта,
всегда сначала dry-run, и корректно останавливается на драфте, если трекер не
настроен. Story Points по Фибоначчи (`1/2/3/5/8/13`) — **опциональная
аннотация, не гейт**: разбиение ведут SPIDR/вертикальные срезы/зависимости.
MD-драфт (`docs/decompose/YYYY-MM-DD-<epic>.md`) — самодостаточный основной
артефакт; пуш в трекер (YouTrack и другие адаптеры) — типовое опциональное
дополнение.

Вызов: `/decompose` (или «нарежь на задачи», «декомпозируй эпик/фичу», «разбей
проект на задачи»). Каждый полученный `<TASK-ID>` дальше подхватывает `task`.

### `task` — пофазовый quality-flow
Проводит один тикет от разбора до закрытия через фиксированные quality-gates:

```
0 ingest → 1 design-spec → 2 premortem(дизайн) → 3 план → 4 premortem(план)
→ 5 TDD → 6 code-review → 6b security-review* → 7 verify live → 8 close
```

`*` условно: security-проход запускается только если диф трогает auth, ввод,
сеть, секреты, data/PII, миграции или гранты. Это **LLM**-слой — gameable и
коррелирует с исполнителем, — поэтому он не заменяет детерминированный гейт, и
наоборот. Одна задача = одна ветка → один MR/PR, CI зелёный (включая гейт) до
merge.

Вызов: `/task DEV-475` (или «прогони через наш flow»). Дисциплина фиксирована;
конкретные команды трекера/VCS/сборки берутся из `CLAUDE.md` проекта.

### `ci-gate` — раскладка детерминированного гейта
Ставит в любой репозиторий переносимый неподкупный «пол» под категории, которые
не ловят ни тесты, ни LLM-ревью:

- **secret-scan** — gitleaks, в pre-commit и CI
- **migration-guard** — инструмент-агностичный, по путям: forward-only
  неизменяемость + деструктивный DDL только с маркером `-- destructive: approved`
- **protected-branch** — запрет force-push + required status checks (правило
  платформы, ставится один раз через `glab`/`gh`)

Вызов: `/ci-gate` в репозитории. Копирует `ci/`, конфиги gitleaks и pre-commit,
CI-файл под GitLab **или** GitHub, печатает команды protected-branch. Полезная
нагрузка шаблона — в `templates/ci-gate/`.

Executor-aware для GitLab: вариант под **docker/k8s** (через `image:`) и вариант
под **shell**, который тянет pinned gitleaks с проверкой sha прямо в джобе — без
docker, без правки раннера, без эскалации через docker-group. Секрет-скан
инкрементальный (диапазон MR / только новые коммиты на push / полная история по
расписанию).

## Установка

```
/plugin marketplace add umar-s/devpowers
/plugin install task-flow@devpowers
```

Дальше — по пайплайну: `/decompose` режет эпик на задачи, `/task DEV-475`
проводит каждую через quality-flow, `/ci-gate` один раз на репозиторий
раскладывает гейт, через который они все проходят на merge.

## Заметки по дизайну
- Гейт детерминирован намеренно: LLM-ревьюер оптимизирует под «зелёно», а не под
  «верно», и делит слепые зоны с исполнителем. Секреты и деструктив — неподкупная
  проверка; логика — LLM-проход.
- `dep/SCA` идёт закомментированным стабом — языко-зависимый, вписывается под репо.
- migration-guard падает **closed** (exit 2), если в CI не может вычислить base
  ref — гейт, который тихо пропускает, хуже отсутствующего.
