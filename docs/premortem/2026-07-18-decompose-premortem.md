## Дыры и принятые решения

### H-001 — `${CLAUDE_PLUGIN_ROOT}` не работает при установке как personal skill (высокая/высокая)
**Угол:** Исполнитель + Допущения.
Task 11 Step 1 предлагал fallback «скопировать в `~/.claude/skills/`» для приёмочного прогона. Personal skill не плагин → `CLAUDE_PLUGIN_ROOT` не определён → все 7 reference-путей ломаются. Хуже — Claude додумает недостающую логику из тощего SKILL.md, ревьюер увидит правдоподобный черновик и поставит PASS. Механизм Read reference по `${CLAUDE_PLUGIN_ROOT}` в репо ни разу не проверен (ci-gate использует переменную только в bash `cp`).
**РЕШЕНИЕ (принято):** приёмочный прогон только через plugin→working-clone. В Task 8 — явный резолв base-path (bash echo `$CLAUDE_PLUGIN_ROOT` / поиск skill-dir) и Read по абсолютному пути, паттерн проверить. В Task 11 pre-check «прочитать один reference через plugin-путь» до dry-run.

### H-002 — «6 полей» захардкожено поверх 8-полевой схемы (высокая/высокая) [кластер: helper1 + helper3]
**Угол:** Исполнитель + Будущий поддерживающий.
Task 4 фиксирует 8 имён (…`truths`, `wave`), но Task 5/6/8/§7/§12 говорят «6 полей» без `truths`/`wave`. Enrich заполнит 6, QA проверит 6 → `truths` (goal-backward, core в GSD) не заполняется и не enforce-ится никогда; `wave` только в таблице. Verify-грепы точечны, ни один не проверяет присутствие всех полей у всех потребителей → будущий ренейм рассинхронизирует 8 референсов молча.
**РЕШЕНИЕ (принято):** единый канон — **6 авторских полей** (name, context, requirements, dod, story_points, depends_on) + `truths` как goal-backward слой ВНУТРИ dod + `wave` computed (не авторское поле, живёт в графе/таблице). Синхронизировать §6/§7/§12, Task 5/6/8. В Task 10 добавить механический тест: каждое имя поля из task-schema.md присутствует у всех потребителей (qa-checklist/draft-template/tracker-sync/SKILL), иначе fail.

### H-003 — Phase 7 (запись в трекер) нетранзакционна и не тестируется до релиза (высокая/высокая)
**Угол:** Противник.
Task 11 останавливается на черновике; необратимый путь (эпик + N подзадач + N связей в боевом трекере) уходит в релиз непрогнанным. Частичный отказ (нет типа «depends», rate-limit, сеть) → orphan-эпик без отката. Идемпотентность по строке summary хрупка: ре-ран с чуть иными summary → полный дубль.
**РЕШЕНИЕ (принято):** добавить **dry-run режим Phase 7** (печать плана создания без записи) — это и безопасность боевого трекера, и тест. Стабильный ключ идемпотентности: маркер `decompose-id:<slug>#<n>` в описании/поле issue, сопоставление по нему, не по summary. Обработка частичного отказа: отчёт «создано X, осталось Y» + идемпотентный ре-ран. Обязательный прогон записи (на черновом эпике / dry-run) ДО Task 12.

### H-004 — SP≤8 как жёсткий гейт атомарности ломает нарезку (высокая/высокая)
**Угол:** Клиент.
SP назначает LLM без калибровки (нет velocity/reference). Но SP сделан управляющим: §6/Global «SP>8 → дробить», QA check 4 заворачивает атомарность по SP≤8, §12 требует «ни одной SP>8». Senior режет крупнее/меньшим числом задач → скилл выдаст рой мелких не по его модели + колонку чисел, которым он не верит → первый черновик в мусор.
**РЕШЕНИЕ (принято):** SP → необязательная аннотация-оценка (юзер выбрал Story Points, оставляем как оценку). Дробление ведём ТОЛЬКО логикой SPIDR / вертикальных слайсов / зависимостей. QA check 4: атомарность = «вертикальный слайс, единый concern, независимо тестируемо»; SP>8 → WARNING (сигнал «посмотри»), не BLOCKER-драйвер ре-сплита. Убрать жёсткое SP-условие из §12.

### H-005 — Публичный плагин, но Phase 7 YouTrack-only + `BA-NNN` захардкожен (высокая/высокая)
**Угол:** Интеграция.
Финальная фаза жёстко на YouTrack MCP + резолв «из CLAUDE.md», которого на свежей установке нет. `task` при этом tracker-agnostic (DEV/JIRA/#123) → пайплайн асимметричен. `BA-NNN` (приватный префикс автора) утёк в return-контракт public repo. Чужой установщик доходит до черновика и молча дохнет на Phase 7.
**РЕШЕНИЕ (принято):** MD-черновик — самодостаточный основной артефакт. Phase 7 = generic tracker-adapter (create-issue/link/estimation через явный контракт); трекер не сконфигурирован → graceful stop на draft с подсказкой. `BA-NNN` → generic `<TASK-ID>`, YouTrack — первый поддержанный adapter (пример, не контракт). Вторая acceptance на ином трекере — опционально (основной артефакт tracker-agnostic).

## Bias-проверка
Наиболее вероятный на топе — **confirmation bias** (план писал автор дизайна, склонен подтверждать свою же архитектуру). Коррекция: приёмочный критерий §12 переведён на независимый прогон + механический cross-consumer тест (H-002) — качество проверяется машиной и свежим субагентом, не автором.
