# Дизайн: `task-flow:decompose`

**Дата:** 2026-07-18
**Автор:** Sergei (umar-s) + Claude
**Статус:** на ревью (brainstorming → spec)
**Репозиторий назначения:** `umar-s/task-flow` (третий скилл рядом с `task` и `ci-gate`)

---

## 1. Назначение (одной строкой)

Скилл берёт крупную единицу работы (свободное описание фичи / существующий эпик `<TASK-ID>` / спека-документ) и **декомпозирует её на хорошо поставленные, связанные между собой задачи** с полным набором полей (Название, Контекст, Требования, DoD, Story Points, зависимости). Выдаёт **самодостаточный MD-черновик для ревью**; по подтверждению — **опциональная заливка в трекер** (generic adapter; YouTrack первым) как эпик + подзадачи со связями. Нет трекера → черновик и есть deliverable.

Стоит **перед** `task-flow:task`. Ядро логики вытянуто из декомпозиционной части GSD (`@opengsd/gsd-core`), без её оркестрового веса.

## 2. Место в пайплайне

```
decompose  →  task (×N)                          →  ci-gate
эпик → нарезанные    каждый <TASK-ID>:               merge-floor
задачи (MD-draft →   spec→2×премортем→TDD→            (secret-scan,
опц. трекер)         code-review→[security-review]→   migration-guard,
                     verify→close                     protected-branch)
```

Связка «GSD режет → твой `/task` исполняет» (обсуждалась 09.07.2026).

## 3. Не-цели (сознательно НЕ тащим из GSD)

Чтобы не получить 68 команд промпт-веса и не задублировать существующий стек:

- **Milestones / roadmap / workstreams** — не нужны, YouTrack эпики этого достаточно.
- **`.planning/`-state, resume-механизм** — у тебя YouTrack (source of truth) + auto-memory + `/clear`-handoff.
- **graphify (AST-граф кодовой базы)** — отдельная тяжёлая машинерия, к нарезке задач не относится.
- **execute/verify/debug-фазы** — это и есть твой `task-flow:task`.
- **Estimation во времени (часов/дней)** — выбрана относительная оценка (Story Points), не FE/BE-часы.

Берём **только декомпозиционное ядро**.

## 4. Вход

Скилл принимает любую из трёх форм (автоопределение по аргументу):

1. **Свободное описание** фичи/проекта текстом.
2. **Ссылка на существующий эпик** YouTrack (BA-NNN) — тогда декомпозирует его на подзадачи.
3. **Спека / дизайн-документ** (путь к MD) — например, выход `superpowers:brainstorming`.

Перед нарезкой всегда читает контекст проекта: `CLAUDE.md`, проектную память (`.claude/projects/.../memory/`), релевантный код.

## 5. Флоу по фазам

| Фаза | Источник в GSD | Что делает |
|---|---|---|
| **0. Ingest & scope** | `questioning`, `edge-probe` | Читает контекст проекта. Если требований нет — вытягивает их (dream-extraction: цель/зачем/для кого/что значит «done»), прогоняет edge-probe по 8 категориям краевых случаев (boundary, empty, encoding, ordering, precision, idempotency, concurrency, adjacency) — **кратко, не интервью**. |
| **1. Requirements** | `requirements.md` | Формирует список `REQ-NN`: user-centric, testable, atomic. Строит traceability «каждое REQ → минимум одна задача». |
| **2. Decompose (ядро)** | `gsd-planner.break_into_tasks`, `spidr-splitting`, `thinking-models-planning` | **Dependency-first, не sequence**: для каждой задачи — что НУЖНО (needs) и что СОЗДАЁТ (creates). **Вертикальные слайсы, не горизонтальные слои**. SPIDR (5 осей: Spike/Paths/Interfaces/Data/Rules) для крупных единиц. Thinking-models: pre-mortem, MECE (каждое требование ↔ ровно одна задача), constraint-first (самое сложное — в начало). **Запрет тихого урезания скоупа** (маркеры «v1/placeholder/stub/basic version» → дробить, а не упрощать). |
| **3. Enrich** | GSD `<task>` + **наше поле SP** | Каждая задача получает полный набор полей (см. §6). |
| **4. Graph & waves** | `depends_on` / `wave` | Граф зависимостей **между задачами** (в GSD только между планами — усиливаем до per-task), проверка ацикличности, вычисление волн параллелизма (`wave = max(deps)+1`), роллап Story Points на эпик. |
| **5. QA нарезки** | `gsd-plan-checker` | **Независимый субагент-чекер** со свежим контекстом и адверсариальной стойкой («считай нарезку дефектной, пока не доказано обратное»). Чек-лист в §7. Цикл ревизий ≤ 3. |
| **6. MD-черновик** | — | Пишет `docs/decompose/YYYY-MM-DD-<epic>.md` (см. §8). **Ждёт апрув/правки пользователя.** |
| **7. Заливка в трекер (опц.)** | — (наше, MCP) | По подтверждению: dry-run → эпик + подзадачи через generic tracker-adapter (см. §9). Нет трекера → стоп на черновике. Возвращает список `<TASK-ID>`, готовых к `task-flow:task`. |

## 6. Схема задачи (поля)

**Канон: 6 авторских полей + `truths` внутри DoD + `wave` (вычисляется).** Все потребители (schema / qa / draft / tracker-sync / SKILL) используют ровно эти имена — единый источник, чтобы один файл не говорил «6 полей», пока другой подразумевает 8.

| Поле | Что содержит | Происхождение |
|---|---|---|
| **name** | action-oriented, глагол+объект | GSD `<name>` |
| **context** | зачем задача, ссылки на код/решения/родственные задачи | GSD `<context>`/`<read_first>` |
| **requirements** | покрываемые `REQ-NN` (непустое) | GSD `requirements` |
| **dod** (4 слоя) | `done` (измеримое состояние) + `acceptance_criteria` (проверяемо) + `verify` (команда <60с) + `truths` (наблюдаемые пользователем факты, goal-backward) | GSD `<done>`/`<acceptance_criteria>`/`<verify>` + `must_haves.truths` |
| **story_points** | Fibonacci 1/2/3/5/8/13 — **необязательная аннотация-оценка, НЕ гейт**; SP > 8 → WARNING «посмотри, может дробить» (дробление ведёт SPIDR/слайсы/зависимости, не SP) | наше (в GSD estimation отсутствует) |
| **depends_on** | список ID задач-предпосылок | GSD `depends_on` (расширено до per-task) |
| **wave** | номер волны параллелизма — **вычисляется из графа (Фаза 4), не авторское поле** | GSD `wave` |

## 7. QA-чек-лист (субагент-чекер, из gsd-plan-checker, сжатый)

Адверсариальная стойка. Каждая находка = severity (blocker/warning). Проверяет:

1. **Покрытие требований** — каждое `REQ-NN` присутствует хотя бы в одной задаче (иначе BLOCKER).
2. **Полнота полей** — у каждой задачи есть все 6 авторских полей, `dod` содержит `truths`, `wave` проставлен (пропуск авторского поля или `truths` → BLOCKER).
3. **Ацикличность графа** — нет циклов, нет ссылок на несуществующие задачи, нет forward-references, волны консистентны с зависимостями.
4. **Атомарность** — каждая задача = вертикальный слайс, единый concern, независимо тестируема. `story_points` > 8 → **WARNING** «посмотри, может дробить», НЕ BLOCKER и не автоматический ре-сплит (дробление ведут SPIDR/слайсы/зависимости).
5. **Связность (key-links)** — артефакты «сшиты», а не созданы изолированно (задача, создающая API, имеет потребителя; UI имеет данные).
6. **Нет тихого урезания скоупа** — скан маркеров упрощения (BLOCKER).
7. **MECE** — нет двух задач на одно требование без чёткого разделения.

Выход: `PASSED` или `ISSUES FOUND` (список с severity + fix-hint). При blocker'ах — ревизия нарезки (≤ 3 цикла).

## 8. Формат MD-черновика

`docs/decompose/YYYY-MM-DD-<epic>.md`:

- **Заголовок эпика** + цель + роллап Story Points + число волн.
- **Таблица задач**: `ID | Название | SP | depends_on | wave | REQ`.
- **Карточки задач**: по одной на задачу со всеми полями из §6.
- **Mermaid-граф** зависимостей (визуализация волн).
- **Traceability-таблица**: `REQ-NN → задачи`.

Пользователь ревьюит, правит, апрувит. Только после апрува — фаза 7.

## 9. Заливка в трекер (generic adapter, MCP)

MD-черновик — самодостаточный основной артефакт. Заливка в трекер — **опциональный дженерик-адаптер**; YouTrack — первый поддержанный адаптер (не контракт). Дженерик-контракт: `create_issue(summary, description, estimate?, parent?)`, `link(from, to, type)`.

По подтверждению (и только по нему):

- **Dry-run по умолчанию** — сперва напечатать полный план создания (summary/description/estimate/связи/стабильный ключ) **без записи**; это и безопасность боевого трекера, и приёмочный тест.
- Создать **эпик** (если вход был описанием/спекой) или использовать существующий `<TASK-ID>`.
- Создать **подзадачи**: `summary` = name; `description` = context + requirements + dod (incl. truths, markdown); estimate = story_points; связи.
- **Стабильный ключ идемпотентности**: маркер `decompose-id:<slug>#<n>` в поле/описании issue; ре-ран сопоставляет по нему, **не по summary** (иначе дубли при чуть иных summary).
- **Частичный отказ**: при падении на середине — отчёт «создано X / осталось Y» с уже созданными id, идемпотентный ре-ран по ключу.
- **Связи**: parent/`subtask of` эпик; `depends on` между задачами по графу.
- **Нет трекера** (MCP не найден и в CLAUDE.md нет привязки) → **graceful stop на черновике**, черновик и есть deliverable (не ошибка).
- Вернуть список созданных/обновлённых `<TASK-ID>`, готовых к `task-flow:task`.

**Маппинг estimate (fallback-лестница):** `story_points` → поле estimate, если есть; иначе Estimation; иначе строка `SP: N` в описании (логировать, что выбрано). Тип связи «depends» — нативный issue-link трекера; если отсутствует — деградация до текстовой пометки в описании с логом.

## 10. Handoff в `task`

Каждый созданный `<TASK-ID>` катится штатным `task-flow:task` (spec → 2×премортем → TDD → code-review → [security-review] → verify → close). `decompose` только готовит хорошо поставленные задачи; исполнение — не его забота.

## 11. Упаковка

- Новый скилл `skills/decompose/SKILL.md` в репо `umar-s/task-flow`.
- Ссылочные материалы (SPIDR, edge-probe категории, thinking-models, QA-чек-лист) — как `references/` внутри скилла, адресуются через `${CLAUDE_PLUGIN_ROOT}`.
- Бамп версии плагина (1.1.0 → 1.2.0), обновить `plugin.json` description и README (RU/EN).
- `marketplace.json` не трогаем (source pinned на `main` → обновится автоматически).
- Локально этот хост обновится через `claude plugin update task-flow` + рестарт.

## 12. Критерии успеха

- На реальном эпике (кандидат: DEV-540 или задача из Makalat) скилл выдаёт нарезку, где: 100% требований покрыты, у всех задач заполнены 6 авторских полей (+`truths` в `dod`), граф ацикличен, QA-субагент возвращает PASSED. `story_points` присутствуют как аннотация; SP > 8 несёт пометку «reconsider» (WARNING, не провал).
- MD-черновик самодостаточен, читается и правится за один проход.
- Phase 7 dry-run печатает корректный план (parent/subtask + depends-связи + estimate + стабильный ключ); реальная запись создаёт корректные `<TASK-ID>` со связями, готовые к `/task`.
